The peer reviewer in our system is a different model from the generator — not a different prompt, a different model — routed to a separate API, with a system prompt designed to refute rather than produce. That difference is the whole point.

Here's what we learned building it.

## The Platform

`vivanti-adp` is a structured knowledge base that Claude Code reads at the start of every session. It encodes how Vivanti delivers data consulting engagements: six agent role definitions, reusable delivery skills, engagement contexts (data contract, repos config, client stack), domain knowledge playbooks, a memory system, and automation hooks. When you open Claude Code inside `vivanti-adp`, it loads your identity, your active engagements, and the routing rules that determine which agent owns which task.

The delivery lifecycle runs in four commands: `discover → init → develop → document`.

`adp-runner` is the command center — a Next.js web app that runs the `develop` step as a streaming multi-agent loop in a browser. You give it a command and an engagement name. It reads context from `vivanti-adp` at runtime — the data contract, the agent definitions, the current tasks list, the accumulated memory — and starts generating artefacts: Snowflake DDL, dbt models, Terraform, CI/CD pipelines.

The output is not a markdown response in a chat window. It is code on a branch with a PR, ready for a human to review and merge. The agent never merges.

## Why Different Models

The main agent is Claude Sonnet 4.6, called via the Anthropic SDK with `client.messages.stream`. It streams tokens to the browser and structures its output with markers parsed client-side:

```
[AGENT: Data Engineer]
[SKILL: schemachange-gen]
[STATUS: aurora_contacts DDL coming up — watching the nulls on phone_number.]
```

The peer reviewer is Gemini 2.5 Flash, called via an OpenAI-compatible LLM gateway.

A reviewer that shares the generator's training distribution is not a reviewer. It will miss what the generator missed, because it missed it for the same reasons. Routing to a different model with a different system prompt changes the failure modes. The peer reviewer's prompt opens with:

> You are a senior data engineer reviewing a teammate's PR. You're direct, you find real problems, and you write like a person — not a system.

Its output is structured for machine parsing:

```
[VERDICT: PASS | PASS WITH NOTES | FAIL]

[CRITICAL] phone_number VARCHAR(15) — exceeds contract maximum of VARCHAR(20)
[MINOR] Missing load_ts audit column — required on all raw tables
```

CRITICAL findings restart the fix loop. MINOR findings surface in the audit log. PASS moves the artefact to handover.

## BlockDetector: Don't Wait for the Full Response

The naive implementation: let the main agent finish, then send everything to the reviewer. Simple, sequential, slow.

A full develop run produces multiple code blocks, each taking 20–30 seconds to generate. Waiting until the last block closes before reviewing the first wastes most of that time.

`BlockDetector` is a line-buffered stream processor that watches for triple-backtick boundaries as tokens arrive:

```ts
class BlockDetector {
  feed(text: string): void {
    const combined = this.lineBuf + text
    const lines = combined.split('\n')
    this.lineBuf = lines.pop() ?? ''
    for (const line of lines) {
      if (!this.inBlock && line.trimStart().startsWith('```')) {
        this.inBlock = true
        this.blockLines = [line]
      } else if (this.inBlock) {
        this.blockLines.push(line)
        if (line.trim() === '```') {
          this.inBlock = false
          this.onBlock(this.blockLines.join('\n'))
          this.blockLines = []
        }
      }
    }
  }
}
```

The instant a block closes, a callback fires an async IIFE that pushes into a `Promise<ReviewResult>[]` array. The main stream continues generating. After the stream finishes, `await Promise.all(pendingReviews)` collects every result.

If the main agent generates five code blocks, blocks one through four are in peer review while the main agent writes block five. On a typical develop run, this saves 30–60 seconds.

## Smart Artefact Naming

Before a block goes to the reviewer, it needs a name — so the reviewer can reference it, the status messages can name it, and the audit log can track it. The `blockName` function tries five extraction strategies in order:

1. **Filename comment on the first content line** — the system prompt requires all code blocks to start with a comment like `-- contacts_raw.sql` or `# pipeline.yml`. This is the primary signal.
2. **SQL DDL** — regex on `CREATE TABLE/VIEW/PROCEDURE` extracts the object name.
3. **SQL DML** — regex on `INSERT INTO` or `MERGE INTO` extracts the target table.
4. **Python** — extracts the first `def` or `class` name.
5. **YAML** — extracts the `name:` key.

If none match, it falls back to `artefact N`. In practice, the filename comment requirement on the system prompt means strategy one fires almost every time. The fallback exists because models occasionally drift from the instruction.

## The Fix Loop: Knowing When to Stop

One review round catches issues. It does not guarantee a fix produces clean code. Fixing one thing can break another. You need iteration.

The loop is an unbounded `while` capped at `MAX_ROUNDS = 5`. On each FAIL verdict, the system identifies which agent should own the fix:

```ts
function agentForBlock(block: string): string {
  const lang = block.match(/^```(\w+)/)?.[1]?.toLowerCase() ?? ''
  const firstLines = block.slice(0, 300).toLowerCase()
  if (firstLines.includes('terraform') || lang === 'hcl') return 'Platform Engineer'
  if (firstLines.includes('{{') || firstLines.includes('ref(') || firstLines.includes('source(')) return 'dbt Engineer'
  if (lang === 'yaml' || lang === 'yml') return 'DevOps Engineer'
  if (lang === 'python' || lang === 'py') return 'Data Engineer'
  return 'Data Engineer'
}
```

The correct specialist fixes their own artefact — a dbt Engineer fixes Jinja templates, not a DevOps Engineer who happened to be last in the queue.

The loop exits on three conditions:

1. **PASS or PASS WITH NOTES** — proceed to handover.
2. **Stale fingerprint** — findings from round N match round N-1. The fix changed nothing meaningful. Escalate to human.
3. **MAX_ROUNDS reached** — five rounds on the same artefact signals an issue the agent cannot resolve from the context it has.

Stale detection normalises and hashes the finding lines:

```ts
function fingerprintFindings(reviewText: string): string {
  return reviewText
    .split('\n')
    .filter(l => /^\[(CRITICAL|MAJOR|MINOR)\]/.test(l.trim()))
    .map(l => l.trim().toLowerCase())
    .sort()
    .join('|')
}
```

A loop that detects its own futility and stops is more useful than one that retries to exhaustion.

## Memory and Idempotency

### PROGRESS.md: Idempotency Across Runs

The main agent emits a fenced `progress-md` block at the end of each develop cycle — a markdown table of every task, its status, the responsible agent, branch name, and notes:

| Task | Description | Agent | Status | Date | Notes |
|------|-------------|-------|--------|------|-------|
| T-001 | Kickoff governance | Platform Engineer | HANDOVER | 2026-01-26 | — |
| T-005 | aurora_contacts DDL | Data Engineer | HANDOVER | 2026-06-26 | branch: data-engineer/aurora-contacts |

The runner extracts this block from the output, appends any recurring-patterns section, and writes it to `engagements/{name}/artifacts/PROGRESS.md`. On the next run, it's loaded server-side and injected into the main agent's context. The agent reads it, sees `HANDOVER`, and skips those tasks.

Run the same engagement twice and you get incremental progress, not duplicated work.

### promoted-patterns.md: Cross-Session Learning

Every review round generates findings. The runner extracts the category from each finding — what it's about, stripped of the specific value — and tallies them across all artefacts in the run. If a category appears in two or more artefacts in a single run, it surfaces in `PROGRESS.md` as a recurring pattern.

If that pattern appears across three or more sessions, it gets promoted to `memory/patterns/promoted-patterns.md`. That file loads into every future run's context as a checklist the main agent reads before generating anything.

The pattern memory is the closest thing we have to institutional knowledge that survives across sessions and engagements.

## What the Agents Sound Like

The system prompt defines a Voice section that applies to prose commentary only — not to code, not to structured markers:

> Write like an engineer typing in Slack. Short sentences, no fluff. Never: "I will now", "As requested". Use: "done", "watch the nulls here", "LGTM."

Each agent has a distinct personality baked into the prompt. The Data Engineer is pragmatic, mildly skeptical of messy data. The dbt Engineer is detail-obsessed. The Platform Engineer treats infrastructure cost as a first-class concern. The Peer Reviewer is encouraging but does not soften findings.

The tone change was not cosmetic. Early output read like a formal audit report. Nobody wants to read that in a consulting context. The Slack-style voice makes the output feel like a team communicating, not a system generating.

After each run, the runner parses the `[AGENT:]` markers, combines them with the peer review entries, and appends a timestamped block to `slack_channel.md`:

```
## Run: 2026-06-26 14:30

[Data Engineer]
Starting on T-005. Aurora contacts DDL coming up.

[Peer Reviewer]
Solid work, but two issues:
[CRITICAL] phone_number VARCHAR(15) — exceeds contract maximum
[MINOR] Missing load_ts audit column

[Data Engineer]
Fixed both. Pushing.

[Peer Reviewer]
LGTM. contacts_raw.sql — all good.
```

Append-only. Every run adds a new block. You end up with a complete audit trail of every decision, finding, and handoff — without reading raw model output.

## What You Should Build Next

### 1. Actual Git Integration

The agent talks about raising a PR. The runner doesn't do it. Wiring in real git operations — branch creation, commits, Bitbucket/GitHub API calls — closes the loop from "agent describes a branch" to "PR exists for human review." This is the highest-value gap.

### 2. Jira / Linear Integration

The develop command instructs the agent to log an issue after every step. It describes what the Jira ticket would say. It doesn't create one. An MCP server or REST client for Jira would give every cycle a real audit trail in your project tracker, not just in `PROGRESS.md`.

### 3. Validator Integration

Peer review is model-based. Models are not interpreters — `sqlfluff` catches style errors, `dbt parse` catches broken `ref()` calls, `terraform validate` catches bad HCL. Running a validator between generation and peer review, then passing its output as additional context to the reviewer, would surface real failures before the model review even starts.

### 4. Cost Tracking

Every run consumes tokens across two models with no visibility into how much. A `$cost` summary line in `PROGRESS.md` — input tokens, output tokens, rough dollar cost — would make this production-usable in a consulting context where token spend maps directly to delivery margin.

### 5. Multi-Engagement Parallelism

One engagement per browser tab. A proper job queue with per-engagement run history would make this a usable tool for a team running three concurrent client engagements.

### 6. Human-in-the-Loop Approval Gate

"Stop" is currently a convention in `PROGRESS.md` — the agent marks tasks `HANDOVER` and stops generating. There is no actual pause mechanism. An explicit approval gate — "PR raised on aurora/T-005. Review and approve before the next task." — would make the human decision point trackable, not implied.

### 7. Promoted-Pattern Automation

Promoting a pattern from `PROGRESS.md` to `promoted-patterns.md` is manual right now. An agent that reads episodic memory across recent runs, identifies categories appearing three or more times, and surfaces them for human promotion would close the learning loop without needing someone to go looking for it.

## The Honest Reflection

We built this in a few weeks of evenings and weekends. It runs real delivery cycles on a real client engagement. The peer review catches genuine issues. The fix loop converges. The idempotency mechanism prevents duplicate work. The audit log is actually useful.

It is not production software. Error handling is minimal. There is no auth. The UI has one hardcoded default engagement. The memory system is flat markdown files read at startup. The git and Jira integrations are theatrical.

That's fine. A prototype's job is to answer whether the architecture is worth building properly. The answer here is yes. Cross-model adversarial review finds things a single model misses. Concurrent review against a stream saves real time. Stale fingerprint detection knows when to stop. Pattern memory accumulates across sessions.

The answer to the headline question — what happens when you make agents review each other's work — is: you catch what a single model misses, you build an audit trail that reads like a team, and you create a feedback loop that gets marginally smarter every run.
