The gap between "an LLM can write a dbt model" and "an LLM can reliably ship a production data platform" is enormous. This is how we started closing it.

## The Problem

Every data consultancy runs the same playbook. You win a client, sign a statement of work, spend the first two weeks writing a data contract, then spend the next eight writing Snowflake DDL, dbt models, Terraform, CI/CD pipelines, and documentation. Most of that work is skilled but heavily patterned. You've done it before. You'll do it again.

We kept asking: what if the scaffolding could write itself?

Not via GitHub Copilot completing a line. Not via Claude drafting a boilerplate SQL file you paste into VS Code. Actually running a delivery workflow — picking the next task, generating the artefacts, peer-reviewing them, fixing the issues, and logging everything — while a human reviews the output.

We built ADP Runner to answer that question.

## What ADP Runner Is

ADP Runner is a Next.js application that takes a command (`develop`, `init`, `discover`) and an engagement's context files — a data contract, a context document, a repos index — and runs a streaming multi-agent loop that produces data engineering artefacts ready for human review.

It is not a chatbot. It is not a code assistant. It is a delivery engine with a peer review loop, a memory system, an audit log, and an idempotency mechanism. The output is code on a branch with a PR, not a markdown response in a chat window.

The UI is intentionally minimal: a command panel, a sidebar showing which agent is active, and a ticker at the bottom. The complexity is in the backend.

## Why We Didn't Just Use Claude Code

Claude Code is excellent at interactive problem-solving. It explores the codebase, it makes changes, you review and redirect. That loop works beautifully for open-ended engineering tasks.

But data consulting delivery is not open-ended:

- Every artefact must conform to a data contract.
- Every table must follow naming conventions, have audit columns, be idempotent.
- Every piece of code should be reviewed before someone signs off on it.
- Every task should be logged, tracked, and handed over — not merged by the agent itself.

Claude Code hands control to you between each step. We wanted something that could run the full cycle — generate, review, fix, hand over — without pausing to ask. Run.

The other reason: we wanted a team. The agent writing Snowflake DDL should not be the same agent reviewing it. A Platform Engineer who provisions Terraform should not be the same agent writing dbt models. Mixing responsibilities makes quality indistinguishable from a single distracted generalist.

## The Architecture

The backend is a single Next.js API route — about 540 lines. No database, no queue, no microservices. Just a streaming response and a set of well-designed functions.

### The Main Agent: Claude Sonnet 4.6 via SSE

The main agent is Claude Sonnet 4.6, called via the Anthropic SDK with `client.messages.stream`. The response is piped into a `ReadableStream` that sends events to the browser. The browser renders tokens as they arrive.

The `SYSTEM` prompt defines the team:

```
Valid agents: Platform Engineer, Data Engineer, dbt Engineer,
              DevOps Engineer, Technical Writer, Peer Reviewer
```

Agents use structured markers to signal transitions:

```
[AGENT: Data Engineer]
[SKILL: schemachange-gen]
[STATUS: T-005 picked up — aurora_contacts DDL]
```

These markers are parsed client-side to show which agent is active, which skill is loaded, and what the current status is. The UI highlights the active agent in the sidebar.

### The Peer Reviewer: Gemini 2.5 Flash

The peer reviewer runs on a separate model — Gemini 2.5 Flash via an OpenAI-compatible LLM gateway. This is a deliberate architecture decision. A reviewer that is adversarial by design should not share the generator's assumptions, quirks, or blind spots. Routing it to a different model with a different system prompt helps.

The reviewer's output follows a structured format:

```
[VERDICT: PASS | PASS WITH NOTES | FAIL]

[CRITICAL] phone_number VARCHAR(15) — exceeds contract maximum
[MINOR] Missing load_ts audit column
```

### BlockDetector: Async Review as Blocks Close

This is the most interesting piece of engineering in the project.

The main agent streams text token by token. We want to fire a peer review the moment a code block closes — not after the full response finishes. A full response can be long and take 30 seconds. So we built `BlockDetector`: a line-buffered stream processor that watches for triple-backtick boundaries and fires a callback the instant a block closes.

```ts
class BlockDetector {
  feed(text: string): void {
    const combined = this.lineBuf + text
    const lines = combined.split('\n')
    this.lineBuf = lines.pop() ?? ''
    for (const line of lines) {
      if (!this.inBlock && line.trim().startsWith('```')) {
        this.inBlock = true
        this.blockLines = [line]
      } else if (this.inBlock) {
        this.blockLines.push(line)
        if (line.trim() === '```') {
          this.inBlock = false
          this.onBlock(this.blockLines.join('\n'))
        }
      }
    }
  }
}
```

Each block fires an async IIFE that pushes into a `Promise<ReviewResult>[]` array. The main stream continues. The reviews run concurrently. After the main stream finishes, we `await Promise.all(pendingReviews)` to collect results and write them to `PROGRESS.md`.

This means if the main agent generates five code blocks, the first four reviews are running in parallel while the main agent is still writing the fifth one. On a typical develop run, this saves 30–60 seconds.

## The Fix Loop

The naive approach: review once, emit findings, done.

The problem: a single round of review often finds issues, but fixing those issues sometimes introduces new ones. Or fixes some and misses others. You need iteration.

We replaced a fixed 2-round loop with an unbounded `while` loop capped at `MAX_ROUNDS = 5`. The loop exits when:

1. **PASS or PASS WITH NOTES**: the reviewer is satisfied — proceed.
2. **Stale fingerprint**: the findings from round N are identical to round N-1. The fix didn't change anything meaningful. This is a human problem. We escalate.
3. **MAX_ROUNDS reached**: five rounds on the same artefact is a signal the issue is ambiguous or requires context the agent doesn't have.

Stale detection works by normalising findings and comparing a hash:

```ts
function fingerprintFindings(reviewText: string): string {
  return reviewText
    .split('\n')
    .filter(l => /^\[(CRITICAL|MAJOR|MINOR)\]/.test(l.trim()))
    .map(l => l.trim().toLowerCase())
    .sort()
    .join('|')
}
```

If the fingerprint matches the previous round's fingerprint: escalate to human, don't retry.

## Memory and Idempotency

Two problems that took longer to get right.

### PROGRESS.md: Idempotency Across Runs

The first time you run `/develop aurora`, the agent creates the task list. What happens when you run it again? Without idempotency, it restarts from scratch — duplicating work, raising duplicate PRs, wasting tokens.

`PROGRESS.md` is the solution. After each cycle, the agent emits a fenced `progress-md` block with a table of every task, its status, the responsible agent, and any notes. The runner writes it to `engagements/{name}/artifacts/`. On the next run, the runner loads it server-side and injects it into the agent's context. The agent reads it, sees which tasks are marked `HANDOVER`, and skips them.

| Task | Description | Agent | Status | Date | Notes |
|------|-------------|-------|--------|------|-------|
| T-001 | Kickoff governance | Platform Engineer | HANDOVER | 2026-01-26 | — |
| T-005 | aurora_contacts DDL | Data Engineer | HANDOVER | 2026-06-26 | branch: data-engineer/aurora-contacts |

### promoted-patterns.md: Cross-Session Learning

Every peer review generates findings. Most findings are one-off. But some findings repeat — across artefacts, across runs, across engagements — nullable foreign keys, non-idempotent DML, hardcoded schema names.

We capture finding categories per artefact. If a category appears in 2 or more artefacts in a run, it surfaces in `PROGRESS.md` as a recurring pattern. If patterns appear in three or more sessions, they get promoted to `memory/patterns/promoted-patterns.md`.

That file is loaded into every run's context as a checklist. The agent reads it before generating anything. It's the closest thing to institutional memory we have right now.

## The Slack-Channel Persona System

This one is less architecture and more craft.

The early agent output read like a formal audit report: "I have reviewed the artefacts and identified the following issues. Upon analysis..." Nobody wants to read that. It doesn't feel like a team; it feels like a system.

We added a Voice section to the `SYSTEM` prompt that takes effect for prose comments, not structured markers:

> Write like an engineer typing in Slack. Short sentences, no fluff. Never: "I will now", "As requested". Use: "done", "watch the nulls here", "LGTM."

Each agent has a personality:

- **Data Engineer**: pragmatic, slightly skeptical of messy data. "Pipelines look clean but I'd add an index on user_id."
- **dbt Engineer**: detail-obsessed. "ref() not source() here. Need a schema test on this column."
- **Platform Engineer**: infrastructure owner. "Costs stay flat."
- **Peer Reviewer**: encouraging but thorough. "Solid overall, but caught one thing:"

The peer reviewer prompt was rewritten tone-first. The formal "You are an independent code reviewer" opening was setting the wrong register. Replacing it with "You are a senior data engineer reviewing a teammate's PR. You're direct, you find real problems, and you respect the person" made an immediate difference.

### slack_channel.md: The Audit Log

After each run, the runner parses the `[AGENT:]` markers, combines them with peer review messages captured inside the review loop, and appends a formatted block to `engagements/{name}/artifacts/slack_channel.md`.

The result reads like a thread:

```
## Run: 2026-06-26 14:30

[Platform Engineer]
T-001 done — contract signed, repos defined, DEV env confirmed. Nothing blocking.

[Data Engineer]
Starting on T-005. Aurora contacts DDL coming up.

[Peer Reviewer]
Solid work, but two issues:
[CRITICAL] phone_number VARCHAR(15) — exceeds contract maximum
[MINOR] Missing load_ts audit column

[Data Engineer]
Fixed both. Pushing.

[Peer Reviewer]
LGTM. contacts_raw.sql — all good.
```

It's append-only. Second run appends a new `## Run:` block. You end up with a chronological record of every decision, finding, and handoff that you can audit without reading raw LLM output.

## What You Should Build Next

### 1. Actual Git Integration

Right now the agent talks about raising a PR. It describes the branch it would create, the files it would commit. The runner does not actually do it — it emits the instructions.

Wiring in real git operations (branch creation, commits, Bitbucket/GitHub API) would close the loop from "agent proposes" to "PR exists for human review." This is the single highest-value missing piece.

### 2. Jira / Linear Integration

The develop command instructs the agent to log an issue after every step. It doesn't. Wiring in an MCP server or REST client for Jira would give every cycle a real audit trail in your project tracker — not just in `PROGRESS.md`.

### 3. Real Validator Integration

The peer review is model-based. Models are not code-aware — `sqlfluff` catches style, `dbt parse` catches broken refs, `terraform validate` catches bad HCL. Adding a validation step between generation and peer review — run the validator, pass the output as context to the reviewer — would reduce false PASS verdicts.

### 4. Cost Tracking

Every run consumes tokens across two models. Right now there's no visibility into how much a develop run costs. Adding a `$cost` line to the `PROGRESS.md` summary — token counts in, rough cost out — would make this production-usable in a consulting context where cost is a direct line to margin.

### 5. Multi-Engagement Parallelism

The runner handles one engagement at a time. If you're running three client engagements, you run three browser tabs. A proper job queue with per-engagement run history would make this a proper tool rather than a dev prototype.

### 6. Human-in-the-Loop Approval Gate

Currently the agent hands over and stops. But "stop" is just a convention in `PROGRESS.md` — there's no real pause mechanism. Adding an approval prompt — "The agent has raised a PR on aurora. Review and approve to continue." — would make the handover explicit and trackable.

### 7. Promoted-Pattern Automation

Right now promoting a pattern from `PROGRESS.md` to `promoted-patterns.md` is manual. An agent that reads the episodic memory across recent runs, identifies patterns that have appeared three or more times, and proposes a promotion — with a human approval step — would close the learning loop without human curation.

## The Honest Reflection

We built ADP Runner in a few weeks of evenings and weekends. It works well enough to run real delivery cycles on a real client engagement. The peer review catches genuine issues. The idempotency mechanism genuinely prevents duplicate work. The Slack-channel voice makes the output readable instead of painful.

What it is not is production software. The error handling is minimal. There's no auth. The UI has one hardcoded default engagement. The "memory" system is flat markdown files read at startup. The git and Jira integrations are theatrical.

But that's fine. The point of a prototype is to answer whether the architecture is worth building. The answer here is yes — the loop works, the quality is usable, and the friction is low enough that a consultant would actually run it rather than fight it themselves.

The next step is productionising the loop. Starting with real git operations. Everything else follows from that.
